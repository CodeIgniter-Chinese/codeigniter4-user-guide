# CodeIgniter 文档译文审查报告（详细模板）

> 本文件只定义**报告结构**与**条目模板**；审查流程、问题分级和修订闭环以 `SKILL.md` 为准。需要完整示例时，参考 `examples/review-report-example.md`。

---

## 使用原则

- 报告反映**静态审查**结果；除非真的执行过命令，否则不要声称“已构建”“已运行”“已验证可执行”
- 报告的目标是驱动修订：问题必须**可定位、可替换、可复验**
- 不要把 `SKILL.md` 的整套流程原文搬进报告；报告只记录本次审查发现了什么、建议怎么改

---

## 标准骨架

```markdown
# CodeIgniter 文档译文审查报告

**文件**：`<文件路径>`
**审查时间**：<YYYY-MM-DD HH:MM>
**审查阶段**：ci-translate 审查
**备份文件**：`<文件路径>.en.bak`

## 1. 总体结论
## 2. Critical
## 3. Major
## 4. Minor
## 5. 术语一致性快照
## 6. 格式与排版扫描摘要
## 7. 审查覆盖率统计
```

如确有必要，可追加：

- `## 8. 快速修复建议`
- `## 9. 人工复核建议`

---

## 1. 总体结论模板

```markdown
## 1. 总体结论

- **质量等级**：[✅ 可发布 / ⚠️ 需修订后发布 / 🚫 阻断发布]
- **分项评分**：准确性 X/5 | 术语一致性 X/5 | 中文可读性 X/5 | 格式合规 X/5
- **核心判断**：<一句话概括是否可进入定稿>
- **最重要的 3 个风险点**（若有）：
  1. ...
  2. ...
  3. ...
```

评级建议：

- `✅ 可发布`：无 Critical / Major
- `⚠️ 需修订后发布`：无 Critical，但有 Major
- `🚫 阻断发布`：存在 Critical

## 2–4. 问题条目模板

### 空段落写法

```markdown
> ✅ **未发现 Critical 级别问题**
```

```markdown
> ✅ **未发现 Major 级别问题**
```

```markdown
> ✅ **未发现 Minor 级别问题**
```

### Critical / Major 通用条目模板

```markdown
### [C-1 / M-1] <简短标题>
- **问题类型**：<信息遗漏 / 格式破坏 / 术语不一致 / 翻译腔 / 排版错误 / 逻辑不清>
- **位置**：<章节/小节/可定位片段>
- **原文**：
  ```rst
  <英文原文片段>
  ```
- **译文**：
  ```rst
  <对应中文片段>
  ```
- **问题**：<准确说明漏了什么、错了什么、为什么有问题>
- **建议**：
  ```rst
  <可直接替换的修正译文或修复方式>
  ```
- **影响**：<会误导理解 / 破坏 reST / 降低专业性 / 影响阅读>
- **复验建议**：<修后应如何确认，例如重新运行 guard、重新核对限定词等>
```

### Minor 简化条目模板

```markdown
### [m-1] <优化建议标题>
- **位置**：<段落/片段>
- **当前译文**：
  ```rst
  <当前译文>
  ```
- **优化建议**：
  ```rst
  <优化后的译文>
  ```
- **理由**：<为什么这样更简洁 / 更自然 / 更清晰>
```

使用建议：

- `Critical` / `Major` 保持完整字段，便于修订
- `Minor` 可以略写，但不要只写抽象评价，仍要给出可替换建议
- 同类问题很多时，可在小节开头先给一句汇总，再列代表性样例

## 5. 术语一致性快照模板

```markdown
## 5. 术语一致性快照

> ✅ **术语使用一致** - 未发现术语混用问题
```

或：

```markdown
## 5. 术语一致性快照

| 英文术语 | 标准译法 | 当前译法（位置） | 结论 |
|---------|---------|----------------|------|
| helper | 辅助函数 | "帮助函数"（第 3 节） | ❌ 需统一 |
| migration | 迁移 | "迁移文件"（第 5 节，多处概念语境） | ⚠️ 按语境调整 |
```

可选补充：

- 高频术语统计
- 是否适合批量修复
- 哪些术语需要人工按语境判断

---

## 6. 格式与排版扫描摘要模板

```markdown
## 6. 格式与排版扫描摘要

- **reST 指令 / 交叉引用 / 锚点**：✅ 全部保留正确 / ⚠️ 发现 X 处问题
- **标题装饰线**：✅ 全部符合规则 / ⚠️ 发现 X 处长度不足
- **代码块 / 行内字面量**：✅ 未翻译 / ⚠️ 发现 X 处问题
- **中英文间距**：✅ 符合规范 / ⚠️ 发现约 X 处需调整
- **数字与单位**：✅ 符合规范 / ⚠️ 发现约 X 处需调整
- **标点与专有名词大小写**：✅ 符合规范 / ⚠️ 发现约 X 处问题
```

这一节适合做扫描摘要，不必重复写成长篇问题描述；具体样例放回 Critical / Major / Minor。

```markdown
## 7. 审查覆盖率统计

- **审查段落总数**：<N>
- **发现问题段落**：<M>
- **问题段落占比**：<M/N = %>
- **检查的 reST 指令数**：<N>
- **检查的代码块数**：<N>
- **检查的交叉引用数**：<N>
- **检查的术语出现次数**：<N>
```

如需更细，可以追加问题分布表：

```markdown
| 问题类型 | Critical | Major | Minor | 总计 |
|---------|---------|-------|-------|------|
| 格式破坏 | X | Y | Z | N |
| 信息遗漏 | X | Y | Z | N |
| 术语不一致 | - | X | Y | N |
| 翻译腔 | - | X | Y | N |
| 排版错误 | - | X | Y | N |
| **总计** | **X** | **Y** | **Z** | **N** |
```

## 8. 快速修复建议（可选）

仅在存在**明显机械性问题**时输出本节，例如：

- 同一术语需要全文统一
- 明显的中英文空格问题可批量修复
- 某类标点问题可半自动处理

如果问题涉及语义、上下文或可能误伤代码块 / 字面量，就不要给机械批量命令，改写成“建议人工逐段修复”。

模板：

```markdown
## 8. 快速修复建议

1. **必须优先修复**：<Critical 问题摘要>
2. **建议随后修复**：<Major 问题摘要>
3. **可延后优化**：<Minor 问题摘要>
```

---

## 9. 人工复核建议（可选）

适用于以下情况：

- 术语需要按上下文判断
- 中文重写可能存在“是否删减信息点”的边界疑问
- 需要实际运行 `make html` 或示例代码才能确认

模板：

```markdown
## 9. 人工复核建议

1. **术语选择**：<哪里需要结合上下文判断>
2. **信息点边界**：<哪里需要确认是否过度改写>
3. **运行验证**：建议本地执行 `make html`，确认格式修复是否引入新问题
```

```

## 设计意图

本模板刻意把“报告骨架”和“问题条目模板”分开：

- 骨架保证输出结构稳定
- 通用条目模板避免为 Critical / Major / Minor 各写一套重复说明
- 可选章节只在确有需要时出现，避免报告过长但信息密度低
