CodeIgniter 用户指南（Sphinx + reStructuredText）翻译工具，内置质量保障流程。

**工作方式**：
- 备份 → 翻译 → guard → 审查 → 按审查结果修订 → guard → 定稿 → 删除备份

**使用方法**：
```
/ci-translate source/libraries/caching.rst
```

- 只接受一个参数：待翻译的 `.rst` 文件路径

# 完整流程（翻译 + 审查）

## 1. 备份原文件

```bash
cp $FILE_PATH $FILE_PATH.en.bak
```

## 2. 翻译内容

读取 `$FILE_PATH.en.bak`（英文 .rst 文件），按照以下硬约束翻译为中文：

### 翻译过程检查点

#### 检查点 1：翻译前准备（翻译五步法第1步后）
- [ ] 已通读完整段落/章节
- [ ] 已提取所有关键信息点（Critical类）
- [ ] 已标注术语（是否需要翻译/如何翻译）
- [ ] 已识别长句（>25词）并规划拆分策略

#### 检查点 2：首段翻译后自检
- [ ] 朗读译文，无生硬感
- [ ] 信息点逐条核对，无遗漏
- [ ] 代词已删除/有充分保留理由
- [ ] 术语译法正确

#### 检查点 3：每个章节翻译后
- [ ] 运行简化版 guard（仅检查该章节的指令数量）
- [ ] 标题装饰线长度检查
- [ ] 术语一致性自查（该章节内）

#### 检查点 4：全文翻译完成后
- [ ] 运行完整 guard 验证
- [ ] 术语全文一致性检查
- [ ] 中英文间距批量检查
- [ ] 代词全文扫描（搜索"你/你的/你们"）

#### 检查点 5：提交前最终确认
- [ ] 朗读全文（或关键章节）
- [ ] 对照原文做最后一次信息点对账
- [ ] 确认 guard 通过
- [ ] 确认审查报告无 Critical/Major 问题

### 翻译核心原则

**先理解，后翻译；重意合，轻形合**

翻译前必须彻底理解原文含义，然后用符合中文阅读习惯的方式重新组织句子，确保译文既保持原意、又流畅自然。

### A. reST/Sphinx 结构零破坏

以下内容不得翻译、不得改写、不得改动目标/参数、不得增删空格或缩进：

1. **指令与块标记**：硬性规定：指令数量和类型必须与原文完全一致，既不能删除，也不能添加

   **警告类指令（admonitions）**：`.. attention::`、`.. caution::`、`.. danger::`、`.. error::`、`.. hint::`、`.. important::`、`.. note::`、`.. tip::`、`.. warning::`、`.. admonition::`、`.. seealso::`

   **代码指令**：`.. code-block::`、`.. literalinclude::`、`.. highlight::`、`.. sourcecode::`、`.. code::`

   **内容指令**：`.. contents::`、`.. toctree::`、`.. include::`、`.. index::`、`.. glossary::`、`.. rubric::`、`.. raw::`、`.. class::`、`.. role::`、`.. table::`、`.. csv-table::`、`.. list-table::`、`.. image::`、`.. figure::`、`.. topic::`、`.. sidebar::`

   **版本指令**：`.. versionadded::`、`.. versionchanged::`、`.. deprecated::`、`.. versionremoved::`

   **Sphinx 特有**：`.. centered::`、`.. hlist::`、`.. productionlist::`、`.. tabularcolumns::`、`.. math::`、`.. only::`、`.. sectionauthor::`、`.. codeauthor::`

   **PHP 域指令**：`.. php:namespace::`、`.. php:class::`、`.. php:method::`、`.. php:function::`、`.. php:attr::`、`.. php:trait::`、`.. php:interface::`

2. **交叉引用与角色**：`:doc:` `:ref:` `:class:` `:meth:` `:func:` `:term:` 等角色名与目标保持英文原样，仅翻译可见文本（若存在）
3. **行内标记与转义**：`*emphasis*`、`**strong**`、``literal`` 的定界符必须保留；反斜杠转义（`\*`）必须原样保留
4. **显式目标/锚点/替换**：`.. _label:`、`|substitution|`、脚注与引用标记等标识符不得翻译
5. **缩进与空行**：段落、列表、代码块、表格的对齐必须保持，不要"顺手格式化"

### B. 代码与字面量零翻译

以下内容一律保持英文原样：

- 代码块、命令行示例、配置片段、输出日志
- 行内字面量（``like this``）中的内容
- 类名/命名空间/方法/属性/常量/配置键/文件路径/URL/HTTP 头字段名/状态码/参数名

### C. 标题装饰线规则

- 章节标题文本需要翻译
- 装饰符号（`=` `-` `~` 等）必须保持同一种符号
- 装饰线长度必须 ≥ 标题文本长度；若中文标题更长，允许把装饰线加长（优先保持不变）

## 3. 术语与译名（以现有中文用户指南为准）

### 固定不翻译（永远保留英文）

- 框架组件/类/接口/特性名：`Session`、`Email`、`Model`、`Entity`、`Config` 等
- 方法/函数/属性/常量/配置键：`find()`、`where()`、`APPPATH`、`database.default.hostname` 等

### 标准译法（强制一致）

| 英文 | 中文 | 上下文 |
|------|------|--------|
| helper | 辅助函数 | |
| library | 类 | |
| controller | 控制器 | |
| model | 模型 | |
| view | 视图 | |
| route/routing | 路由 | |
| filter(s) | 过滤器 | |
| migration(s) | 迁移 | |
| seeder(s) | 数据库填充 / 数据填充 | 按上下文选择 |
| validation | 验证 | |
| HTTP header | HTTP 标头 | |
| HTTP verb / method | HTTP 方法 | HTTP 语境 |
| Auto Routing (Improved) | 自动路由（改进版） | |
| Auto Routing (Legacy) | 自动路由（传统版） | |

### 术语上下文判断规则

#### 判断决策树

**seeder**:
- 上下文包含"run/seeds/generate data" → **数据库填充**（强调过程）
- 上下文包含"create/write Seeder class" → **数据填充**（强调对象）
- 首次出现时，查看后续用法，统一选择

**migration(s)**:
- 作为文件名/类名出现 → **迁移文件**（如"Migration files"）
- 作为概念/过程出现 → **迁移**（如"Run migrations"）
- 标题中优先使用"迁移"

**configuration**:
- 上下文为名词短语 → **配置**（如"configuration file" → "配置文件"）
- 上下文强调过程 → **配置过程**（如"configuration process"）

#### 术语选择四步法

1. **识别语境**：阅读该段落后 2-3 句，判断是强调"过程"还是"对象"
2. **一致性检查**：搜索全文该术语的所有出现位置，判断是否需要统一
3. **首次决定**：文档首次出现该术语时，选择最符合整体语境的译法
4. **标注决策**：在审查报告中标注术语选择及理由（若存在歧义）

## 4. 中文技术文案规范

1. **中英文间距**：中文与英文/数字之间加空格
   - ✅ "CodeIgniter 4"、"HTTP 请求"
   - ❌ "CodeIgniter4"、"HTTP请求"

2. **单位格式**：数字与单位不加空格
   - ✅ "10Gbps"、"50ms"
   - ❌ "10 Gbps"、"50 ms"

3. **中文标点**：中文句子使用全角标点（，。！？；：），不重复标点

4. **中文优先，杜绝翻译腔**：
   - **禁用无意义代词**：彻底删除"你/你的/你们"、"允许你/使你能/让你可以"、"我们"等冗余表达
   - **示例**："This allows you to configure..." → "配置..."（而非"这允许你配置..."）
   - **示例**："You can use this method to..." → "此方法可用于..."（而非"你可以使用此方法..."）
   - **动词化优先**：将"进行/实施/执行 + 名词"改为直接动词，如"进行配置"→"配置"
   - **去冗余**：删除"实际上/事实上/本质上/简单来说/需要注意的是"等填充词

5. **代词策略**：**彻底禁用"你/你的"等人称代词**。中文技术文档以陈述为主，直接说明功能和行为。

#### 代词处理决策树

**第一原则：能删则删**
- ❌ "你可以使用此方法..." → ✅ "可使用此方法..."
- ❌ "它提供了..." → ✅ "提供..."

**可以保留代词的场景**：

1. **对比场景**（需要区分不同主体）
   - ✅ "控制器处理请求，而模型处理数据"（虽然可删，但保留"而"更清晰）

2. **强调场景**（强调特定对象）
   - ✅ "**你自己**需要负责处理异常"（强调主体）

3. **用户视角说明**（确实是在描述用户行为）
   - ✅ "你可能会遇到以下情况..."（描述用户可能的状态）
   - 但更推荐：✅ "可能会遇到以下情况..."

**代词处理检查清单**：
- [ ] 删除代词后，句子主语是否清晰？
- [ ] 删除代词后，句子是否更简洁？
- [ ] 删除代词后，是否影响理解？
- [ ] 如保留代词，是否有充分的理由？

**测试方法：代词删除测试**：
1. 尝试删除代词
2. 朗读删除后的句子
3. 如果删除后更清晰/同样清晰 → 删除
4. 如果删除后影响理解/不自然 → 保留，但需说明理由

## 5. 可读性增强：深度理解 + 中文重构（必须执行）

**核心原则**：彻底理解原文含义后，用中文思维重新组织表达，而非逐句对应翻译。

### 翻译五步法（内部执行，不输出过程）

1. **语义理解**：通读整段/整节，把握核心意图、逻辑关系和语气（陈述/警告/建议/限制）
2. **信息提取**：按信息点分类清单提取关键信息点，忽略英文句式结构
3. **中文重构**：用自然中文重新组织这些信息点：
   - 按中文习惯调整语序（重要信息前置）
   - 将名词化结构转为动词短句
   - 把从句拆分为独立短句
   - 用中文固有表达替换翻译腔
4. **覆盖核对**：逐条确认每个信息点在译文中准确呈现（零遗漏、零增添、零篡改）
5. **润色定稿**：朗读检验流畅度，删除冗余词，统一术语

#### 信息点分类提取清单（翻译时逐项核对）

**A. 必须提取的关键信息点（Critical - 遗漏即阻断）**

- [ ] **版本/时间限定**："4.3 or later" → "4.3 或更高版本"、"before 4.0" → "4.0 之前"
- [ ] **强制性限定词**："must/only/required/not allowed" → 必须/仅/必需/不允许
- [ ] **条件与前提**："if/when/before/after/unless" → 如果/当...时/在...之前/在...之后/除非
- [ ] **安全警告**："Warning/Danger/Do not/Never" → 警告/危险/不要/从不
- [ ] **技术细节**：数值/参数名/返回值类型/版本号/配置键名
- [ ] **否定表达**："not/never/no/none" → 不/从不/无/没有

**B. 重要信息点（Major - 遗漏影响理解）**

- [ ] **功能说明**：动作/对象/目的/结果
- [ ] **建议性限定词**："should/recommended/better/preferred" → 应该/建议/更好/优先
- [ ] **因果关系**："because/so/therefore/as a result" → 因为/所以/因此/结果是
- [ ] **并列关系**："and/or/also/as well as" → 和/或/也/以及
- [ ] **顺序关系**："first/then/finally/next" → 首先/然后/最后/接下来
- [ ] **对比关系**："but/however/while/whereas" → 但是/然而/而/反之

**C. 次要信息点（Minor - 可灵活处理）**

- [ ] **修饰性副词**："very/extremely/slightly/highly" 等（可适度调整或删除）
- [ ] **举例说明**："for example/such as/like/instance" → 例如/如/比如
- [ ] **补充说明**："note that/by the way/incidentally" 等（可融入正文或省略）

**提取方法**：
1. 阅读英文原文时，在心中或草稿上标注每个信息点及其类型（A/B/C）
2. 翻译前确认所有 A 类信息点都已识别
3. 翻译后逐项核对清单，确保无遗漏

### 中英文表达差异对照（关键）

| 英文表达 | ❌ 直译腔 | ✅ 中文习惯 |
|---------|---------|-----------|
| allows you to | 允许你去... | 直接说功能，如"用于...""可..." |
| enables you to | 使你能够... | 删去，直接动词开头 |
| you can | 你可以... | 直接陈述，如"可...""支持..." |
| it is recommended to | 推荐你去... | 建议... / 宜... |
| please note that | 请注意... | 直接陈述要点 |
| in order to | 为了... | 直接说目的，或省略 |
| by default | 默认情况下 | 默认... |
| for example | 举个例子来说 | 例如 / 如 |
| on the other hand | 另一方面 | 反之 / 而 / 或 |

### 可读性目标（不降低技术精度）

- **意合优先**：中文重意合、轻形合，不必保留英文从句结构
- **短句为主**：超过 25 词的英文句，优先拆成 2-3 个中文短句
- **动词开头**：技术说明以动词或功能名词开头，避免"这/它/你"等冗余主语
- **一段一中心**：先说"是什么/结论"，再说"为什么/怎么用"
- **自然流畅**：读出声应顺口，无生硬感，像中国人写的技术文档

### 中文重构质量评估清单

#### 句子层面质量评估（每句自查）

| 维度 | 评估标准 | 不合格示例 | 合格示例 |
|------|---------|-----------|---------|
| **长度** | 单句≤30字；技术句≤25字 | "默认缓存生命周期为60秒，如果需要调整此值可以在配置文件中修改..." | "默认缓存生命周期为60秒。如需调整，在配置文件中修改..." |
| **主语** | 无冗余主语（它/这/你） | "它提供了全面的功能..." | "提供全面的功能..." |
| **动词** | 动词开头或功能名词开头 | "这是一个用于...的方法" | "用于...的方法" |
| **逻辑** | 先结论后细节 | "如果A，那么B，因为C" | "B，因为C。前提是A" |
| **连贯** | 无逻辑跳跃 | "配置文件中修改参数。构建项目。" | "配置文件中修改参数后构建项目" |

#### 段落层面质量评估（每段自查）

- [ ] **主题句**：首句明确段落主题（是什么/做什么）
- [ ] **信息密度**：无空洞表达（"需要注意的是"、"简单来说"）
- [ ] **逻辑顺序**：按"是什么→为什么→怎么用→注意事项"组织
- [ ] **并列结构**：并列项结构对称（都是动词开头/都是名词短语）
- [ ] **转折标记**：有明确的转折标记（然而/但是/反之）

#### 可读性评分卡（每段打分 1-5）

| 维度 | 1-2分（差） | 3分（中） | 4-5分（优） |
|------|----------|---------|----------|
| **简洁性** | 多处冗余表达 | 偶有冗余 | 无冗余，每词有作用 |
| **清晰性** | 需要反复阅读才能理解 | 读一遍能理解 | 一目了然 |
| **流畅性** | 读起来生硬/有翻译腔 | 基本流畅 | 像母语者写的 |
| **专业性** | 术语错误/不统一 | 术语正确但不统一 | 术语准确且统一 |

## 6. 输出前自检（必须执行）

- **完整性**：不漏译、不多译，技术细节保持一致
- **格式**：指令、角色、锚点、缩进、空行、表格对齐、代码块缩进全部保留
- **标题**：符号不变，装饰线长度 ≥ 标题长度
- **术语**：同一概念全篇一致
- **中文流畅度**：朗读译文，无生硬感、无翻译腔、无不自然停顿
- **代词清理**：全文搜索"你/你的/你们/允许你/使你能"，彻底删除或改写
- **信息点对账**：对每段英文至少做一次"要点清单→中文逐条对应"的核对
- **细节哨兵**：凡出现 must/should/only/except/required/not allowed/注意事项、以及数值/版本/参数/返回值，中文必须逐一出现且语气强度匹配

将翻译好的中文内容写入原文件 `$FILE_PATH`（覆盖英文原文）。

## 7. guard 校验

运行专用 guard，检查译文是否破坏关键 reST/Sphinx 结构（会优先对比第 1 步生成的英文备份 `$FILE_PATH.en.bak`）：

```bash
python3 ${CLAUDE_SKILL_DIR}/scripts/validate-rst.py --no-build "$FILE_PATH"
```

如果失败，检查并修复格式问题后重新执行，直至 guard 通过。

## 8. 自动审查（必要步骤）

翻译完成后，立即执行审查流程，生成审查报告。

若报告发现问题：
1. 按照报告建议逐条修订译文
2. 修订后再次运行 guard 验证
3. 重新做一轮审查，直至无 Critical/Major 问题再定稿

## 9. 审查流程

对当前翻译结果进行质量审查，作为定稿前的强制验收步骤。

**前提条件**：必须存在英文备份文件 `$FILE_PATH.en.bak`，并以其作为英文对照源。

审查时不要把它当成独立模式，也不要只输出审查报告后结束；审查的目标是驱动修订并完成最终定稿。

## 审查核心方法：信息点对账 + 可读性重写建议

### 1) 信息点对账（Accuracy without loss）

对每个段落（或每个列表项/说明块）：

- 从英文中提取"信息点"（事实/条件/限制/步骤/例外/警告/原因/结果/术语定义）
- 核对中文是否逐条覆盖：不漏、不多、不改含义；must/should/only/except 等语气强度必须一致
- 若发现遗漏/增添/弱化/强化：按 Critical/Major 分类，并给出"保留全部信息点"的修订建议

### 2) 可读性与结构审查（Meaning-first rewrite quality）

当中文"绕/难读"时，提出"意义驱动重写"建议：

- **允许**：调整语序、拆分长句、合并同类项、主结论前置、主动语态、动词化表达、段落重组（不破坏 reST 结构前提下）
- **禁止**：删除任何信息点；禁止添加原文没有的解释性内容
- **建议改写必须显式保留原段所有信息点，并尽量更清晰**

#### 可读性常见问题类型

- 主语缺失或指代不明（"它/这/该"无法定位）
- 从句堆叠导致逻辑链过长（建议拆为 2-3 句）
- 名词化/介词化过度（"进行…的…的…"）
- 被动语态过多（可改主动表达但不改变语义责任主体）
- 信息顺序不符合中文阅读习惯（先结论/用途，后细节/原因/注意事项）
- 并列结构不对称、列表句式不统一

### 可读性量化指标与阈值

#### 句子复杂度指标

| 指标 | 阈值 | 测量方法 | 超标处理 |
|------|------|---------|---------|
| **句子长度** | ≤30字（技术句≤25字） | 统计每句字数 | 拆分为2-3句 |
| **从句数量** | ≤2个 | 统计逗号后是否跟完整子句 | 拆分句子 |
| **名词化结构** | ≤1层 | "A的B的C"算2层 | 改为动词短语 |
| **介词堆叠** | ≤3个 | "在A中对B进行C" | 调整语序 |

#### 段落质量指标

| 指标 | 阈值 | 测量方法 | 超标处理 |
|------|------|---------|---------|
| **段落长度** | ≤150字 | 统计段落数字数 | 拆分为多段 |
| **句子数量** | 3-5句 | 统计句号数量 | 重组段落 |
| **信息点密度** | 每句1-2个核心信息点 | 人工判断 | 拆分信息点 |

#### 可读性自动检测建议

审查时可采用以下简化检查方法：

1. **句子长度快速检查**：
   - 目测每句是否超过一行（约30字）
   - 超过则标记为"可能需要拆分"

2. **从句数量快速检查**：
   - 统计每句的逗号数量
   - 逗号超过2个且每部分都是完整句子 → 标记为"从句过多"

3. **名词化结构快速检查**：
   - 搜索"的"字出现次数
   - 单句超过2个"的" → 标记为"名词化过度"

## 格式与技术硬约束（任何破坏都算严重问题）

### reST/Sphinx 结构零破坏（Critical）

以下内容不得翻译、不得改写、不得改动目标/参数、不得随意改空格或缩进：

- **directives 及其选项**：`.. note::`、`.. warning::`、`.. code-block::`、`.. literalinclude::`、`.. toctree::`、`.. include::` 等
- **段落结尾双冒号**：`::` 必须保持英文半角冒号，禁止翻译成中文全角冒号 `：`
  - ❌ 错误：`设置变量如下：：` 或 `设置变量如下：` 或 `设置变量如下:`
  - ✅ 正确：`设置变量如下::`（保持半角双冒号）
- **交叉引用角色与目标**：`:doc:` `:ref:` `:class:` `:meth:` `:func:` `:term:` 等
- **显式目标/锚点/替换标识符**：`.. _label:`、`|substitution|`、脚注标记
- **行内标记与转义**：`*em*`、`**strong**`、``literal``、反斜杠转义
- **表格/列表/代码块缩进与对齐**（不要"格式化"）

### 标题装饰线规则（Critical）

- 标题文本需要翻译
- 装饰符号（`=` `-` `~` 等）必须保持同一种符号
- 装饰线长度必须 ≥ 标题文本长度；中文标题更长时允许加长（优先保持不变）

### 代码与字面量零翻译（Critical）

以下内容一律保持英文原样：

- 代码块、命令行、配置片段、输出日志
- 行内字面量（``...``）内的内容
- 类名/方法名/常量/配置键/路径/URL/HTTP 头字段名/状态码/参数名

## 术语一致性（必须与 translator 对齐）

| 英文 | 标准中文译法 | 上下文 |
|------|--------------|--------|
| helper | 辅助函数 | |
| library | 类 | |
| controller | 控制器 | |
| model | 模型 | |
| view | 视图 | |
| route/routing | 路由 | |
| filter(s) | 过滤器 | CI4 语境 |
| migration(s) | 迁移 | |
| seeder(s) | 数据库填充/数据填充 | 按上下文选择 |
| validation | 验证 | |
| HTTP header | HTTP 标头 | |
| HTTP verb / method | HTTP 方法 | HTTP 语境 |
| Auto Routing (Improved) | 自动路由（改进版） | |
| Auto Routing (Legacy) | 自动路由（传统版） | |

**CodeIgniter 类名、方法名、常量、配置键一律保留英文原样**

## 中文文案排版检查要点

- 中英文/数字之间加半角空格（例：CodeIgniter 4、HTTP 请求）
- 数字与单位不加空格（例：50ms、512MB）
- 中文句子用全角标点（，。！？；：），不重复标点
- 专有名词大小写正确（GitHub、MySQL、API）

## 问题分级

- **Critical**：会导致构建失败/误导技术理解/破坏代码与字面量/破坏 reST 结构/遗漏关键限制与安全提示
- **Major**：术语不一致、明显翻译腔影响理解、排版错误较多、逻辑关系不清
- **Minor**：可读性与风格微调（不影响理解但可更清晰）

### 审查判断标准与阈值

#### 信息点遗漏严重程度判断表

| 遗漏类型 | 示例 | 严重程度 | 判定理由 |
|---------|------|---------|---------|
| 版本/时间限定 | "4.3 or later" → "4.3" | **Critical** | 误导用户，可能导致功能不可用 |
| 强制性限定词 | "must" → "需要" | **Critical** | 改变语义强度，可能误导用户 |
| 安全警告 | "Warning" → "注意" | **Critical** | 降低安全级别，可能导致风险 |
| 功能说明 | "retrieve data" → "获取" | **Major** | 信息不完整，但基本可理解 |
| 条件与前提 | "if configured" → 遗漏 | **Major** | 可能导致用户忽略前提条件 |
| 建议性限定词 | "should" → "可以" | **Minor** | 语气变化，不影响基本理解 |

#### 可读性问题严重程度判断表

| 问题类型 | 示例 | 严重程度 | 判定理由 |
|---------|------|---------|---------|
| 主语缺失导致歧义 | "它提供了...用于..."（无法确定"它"指代） | **Major** | 影响理解 |
| 句子过长（>30字）且逻辑复杂 | 一句包含多个条件/转折 | **Major** | 阅读障碍 |
| 翻译腔明显 | "这允许你去配置..." | **Major** | 不专业，影响阅读体验 |
| 句子稍长（25-30字）但逻辑清晰 | 单句包含2个并列动作 | **Minor** | 可接受，但建议优化 |
| 标点不一致 | 半角标点混用 | **Minor** | 不影响理解 |

#### 术语不一致严重程度判断表

| 情况 | 示例 | 严重程度 | 判定理由 |
|------|------|---------|---------|
| 关键术语错误 | "helper" → "帮助函数"（应为"辅助函数"） | **Major** | 不专业，可能导致混淆 |
| 同一文档内不一致 | 同一文档中"helper"有时译"辅助函数"，有时"帮助函数" | **Major** | 影响一致性 |
| 不同文档间不一致 | 文档A用"辅助函数"，文档B用"帮助函数" | **Minor** | 可接受，但建议统一 |

#### Critical/Major/Minor 边界判定流程

```
问题发现
    ↓
是否会导致构建失败？
    是 → Critical
    否 ↓
是否误导技术理解或存在安全风险？
    是 → Critical
    否 ↓
是否影响阅读理解或术语专业性？
    是 → Major
    否 ↓
是否影响阅读体验或排版规范？
    是 → Minor
    否 → 不算问题
```

## 审查边界（防止编造）

- 只做"静态审查"。**不要声称**"已构建/已运行/已验证可执行"

- 不要编造问题或夸大严重程度
- 遇到不确定的情况，标注为"**需人工复核**"
